# ADR-0001: Framework version and caching model

**Status:** Accepted
**Forced by:** Roadmap M0 (project scaffolding)

## Context

The PRD pins the frontend to "Next.js — current supported stable major at implementation time" and explicitly requires the exact version and caching model to be recorded in an ADR rather than left implicit. M0 is the first point this decision is unavoidable — scaffolding requires an actual version.

## Decision

- **Next.js 16.2.11** (latest stable at scaffold time, 2026-07-23), App Router, `src/` directory layout, TypeScript, Tailwind CSS 4, ESLint.
- **React 19.2.4 / React DOM 19.2.4** (versions Next.js 16 installs by default).
- **Package manager: npm.** No pnpm/yarn/bun available in the target environment at scaffold time; npm is universally available and needs no extra install.
- **Node.js baseline: 24 (current LTS)** in CI (`.github/workflows/ci.yml`); local dev observed on Node 25.9.0. Prisma's stated minimum is 20.19.0+ — 24 LTS is chosen for CI stability, not as a floor.
- **Caching model:** per `ARCHITECTURE.md` → Caching Strategy — ISR/CDN for published editorial content (invalidated by content revision via outbox), short-lived/uncached for search-filter results, provider-specific TTL with `fetchedAt`/`expiresAt` for dynamic external facts (forecasts, permits), no shared cache for user-specific data. Redis is explicitly deferred (not a Phase 1 dependency).

## Consequences

- Next.js 16 has documented breaking changes from prior major versions (see `AGENTS.md` at repo root, generated by `create-next-app`). Before writing App Router code that leans on APIs from training-era knowledge, check `node_modules/next/dist/docs/` rather than assuming Next 14/15 conventions still apply.
- **Known issue, tracked not blocking:** `npm audit` reports 3 high-severity CVEs in Next.js's own bundled transitive dependencies (`postcss`, `sharp` — XSS/arbitrary-file-read in postcss, libvips CVEs in sharp). We are already on the latest published Next.js version; there is no non-breaking upstream fix yet. `npm audit fix --force` would downgrade Next.js to `9.3.3`, which is not a real fix. CI runs `npm audit --audit-level=high` as a non-blocking, informational step (see `ci.yml`) so this stays visible. Re-evaluate on every Next.js patch bump.
- Pinning to npm means CI and local dev must stay on npm; switching package managers later requires a new ADR (lockfile format changes, not a config toggle).
